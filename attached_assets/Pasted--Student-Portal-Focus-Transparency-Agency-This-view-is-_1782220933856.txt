🎓 Student Portal (Focus: Transparency & Agency) This view is designed to give students a fair, transparent environment before their work is officially graded. "Draft Mode" & False-Positive Dispute Pipeline What it is: A student-facing portal where they can submit their assignment for a "Pre-Check" before the final deadline. If it gets flagged as AI, they are prompted to submit a "Dispute Rationale" (e.g., "I used Grammarly for my comments"). Why you need it: AI detectors (even ONNX-based ones) occasionally trigger false positives. If the system is purely punitive, students will rebel. How it works: Gives students agency to correct accidental flags. When the final report goes to the instructor, it includes the student's preemptive defense, saving the instructor from having to initiate an awkward confrontation. 👨‍🏫 Instructor Dashboard (Focus: Day-to-Day Grading & Batch Analysis) This view provides professors and teaching assistants with the tools needed to manage specific assignments, filter out noise, and detect localized cheating rings. The "Skeleton Code" Whitelist (Template Subtraction) What it is: Instructors can upload a "Base Assignment Template" (the starter code provided to the whole class) before students submit their work. Why you need it: Without this, if an instructor gives the class 50 lines of starter code, the system will flag all 40 students for 100% plagiarism because their vectors will perfectly match each other. How it works: The backend vectorizes the instructor's template and mathematically ignores those specific vector chunks when calculating the similarity scores for the student submissions. The Batch Collusion Matrix (Network Graph) What it is: Instead of just comparing one student to the database, this feature compares an entire class's assignment batch against each other and displays a visual network graph. Why you need it: Students often form "cheating rings" where one student writes the code and shares it with three friends. How it works: When an instructor uploads a .zip of the class submissions, the UI (using Vaadin or a lightweight JS library like D3/Mermaid via HTMX) draws a web. If Student A, B, and C share an 85% vector similarity, they are connected by thick red lines, instantly highlighting collaborative cheating. 🏛️ Administrator Console (Focus: Long-Term Oversight & Disciplinary Action) This view is restricted to department heads, deans, and academic integrity committees who need macro-level data and official documentation across a student's entire academic career. Historical Student Profiling (Skill-Spike Detection) What it is: The system tracks a specific student's coding style and AI-usage probability across their entire degree (from Semester 1 to Semester 8). Why you need it: The strongest indicator of cheating isn't just the code itself; it's a sudden, inexplicable jump in skill. If a student struggles to write a basic for loop in Week 3, but submits flawless, enterprise-level microservices in Week 4, it's highly suspicious. How it works: By querying the PostgreSQL database by student_id, the dashboard can show a trendline graph of their "AI Probability Score" over time, alerting administrators to sudden behavioral anomalies. Automated Disciplinary PDF Dossiers What it is: A one-click button that compiles the plagiarism analysis, the visual side-by-side text diff, and the AI probability scores into a branded, printable PDF. Why you need it: If an instructor decides to fail a student or send them to the university disciplinary board, they need hard, printable evidence. A link to a local dashboard won't work for a dean or committee. How it works: Uses a Java library like OpenPDF or iText within the Spring Boot @Service layer. It takes the highlighted chunk data and formats it into an official, standardized university report document.
